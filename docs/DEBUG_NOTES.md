# Blackbox 调试记录与项目复盘

本文档记录 Blackbox 项目开发过程中遇到的主要困难、排查过程和工程收获。  
这些内容是项目最有价值的部分之一，因为它体现的不是“代码能跑”，而是从现象定位到根因、再到修复验证的过程。

---

## 0. 调试总览

本项目遇到的困难不是单一代码错误，而是嵌入式小系统开发中常见的链路问题。项目从单外设实验逐渐扩展到多模块协作，问题也从“某个函数怎么写”升级为“系统链路是否清楚、数据是否可靠、调试证据是否完整”。

主要困难包括：

1. 需求边界不稳  
2. 硬件通信链路不清  
3. 初始化顺序和模块依赖问题  
4. 多文件工程构建问题  
5. `app.c` 调用链复杂  
6. 串口协议升级困难  
7. 时间戳和真实时间来源混淆  
8. MPU6050 检测逻辑调参困难  
9. W25Qxx 日志从“能写”到“可信”  
10. 版本基线不稳  

---

## 1. 需求边界不稳

### 现象

项目最初目标是：

```text
STM32 + MPU6050 + W25Qxx 的异常事件黑匣子
```

后续又不断加入：

- 真实时间
- 上电自动同步现实时间
- HC-05 蓝牙
- PC 上位机
- 微信小程序设想
- GitHub 包装
- README
- 自动刷新状态
- 串口协议升级

### 根因

主线功能还没完全闭环时，支线功能过多，导致当前阶段的验收标准不清晰。

### 收获

以后每加功能先问：

1. 这个功能是不是当前阶段必须做？
2. 它依赖哪些模块？
3. 最小验收标准是什么？
4. 不做它，主项目还能不能成立？

---

## 2. 硬件通信链路不清

### 现象

开发中曾多次混淆：

- HC-05 和串口是什么关系
- CH340 和 HC-05 是否只能用一个
- TX/RX 怎么交叉
- 串口没输出到底是线错、没共地、波特率错，还是代码没跑

### 关键理解

| 模块 | 角色 |
|---|---|
| STM32 USART | 单片机内部串口外设 |
| CH340 | USB 转 TTL 串口，用于有线调试 |
| HC-05 | 蓝牙串口透传模块 |
| PC 上位机 | 接收、解析、显示状态 |

调试链路：

```text
STM32 USART -> TX/RX 接线 -> CH340/HC-05 -> 串口助手/上位机
```

### 收获

串口不通时，不能直接猜。必须分层排查：

1. 供电和共地
2. TX/RX 交叉
3. 波特率
4. STM32 是否发送
5. 外部模块是否透传
6. PC 端是否选对 COM 口
7. 上位机是否正确分帧

## 3. 初始化顺序和模块依赖

### 现象

曾出现：

- OLED 显示异常
- 串口没输出
- 调整初始化顺序后恢复
- 新加模块后旧模块异常

### 根因

嵌入式模块依赖初始化顺序。  
例如串口还没初始化就发送字符串，必然没有输出。

通用链路：

```text
RCC 时钟 -> GPIO 模式 -> 外设初始化 -> 中断/NVIC -> 模块初始化 -> 应用层调用
```

### 收获

遇到“没反应”时，不要先改业务逻辑。先查初始化链路。

## 4. 多文件工程构建问题

### 现象

曾遇到：

- 头文件声明了，`.c` 文件没实现
- 调用了不存在的函数
- `Undefined symbol BlackBoxLog_Clear`
- `App_LogCount` 未定义
- 文件在 VSCode 里有，但没加入 Keil 工程
- 函数名、声明、实现不一致

### 根因

多文件工程需要同时保证：

1. `.h` 里有声明
2. `.c` 里有实现
3. 函数名完全一致
4. `.c` 文件加入 Keil 工程
5. 调用方 include 正确头文件

### 收获

看到 `undefined symbol`，先查：

```text
函数有没有实现？
实现所在 .c 文件有没有加入 Keil？
声明、实现、调用三处名字是否一致？
```

## 5. app.c 调用链复杂

### 现象

曾经不清楚：

- `app.c` 是怎么工作的
- 蓝牙流程调用了哪些函数
- `App_HandleSerialCmd()` 放哪里有没有讲究
- `App_PrintLastLog()` 和 `App_PrintLastLogOnBoot()` 是否一样

### 项目执行链

```text
main 初始化
  ↓
App_Init 初始化业务模块
  ↓
while(1) 调 App_Update / App_Run
  ↓
读取 MPU6050
  ↓
motion_detect 判定事件
  ↓
OLED 显示当前状态
  ↓
异常事件写入 W25Qxx
  ↓
Serial / HC-05 接收命令
  ↓
返回状态 / 查询日志 / 清空日志
```

### 收获

看系统不能只看单个函数，要看：

```text
谁初始化 -> 谁触发 -> 谁更新变量 -> 谁输出结果 -> 怎么验证
```

## 6. 串口协议升级困难

### 现象

早期命令是：

```text
N
L
C
T
```

后期需要支持：

```text
PING\r\n
GET\r\n
LAST\r\n
CLR\r\n
T 2026-04-22 22:17:10\r\n
```

### 根因

单字节接收结构只能可靠保存最后一个字节，不适合长命令。

### 修复

升级为：

```text
裸串口收发 -> 行缓冲 -> 命令解析 -> 文本协议
```

### 收获

串口是字节流。协议必须设计分帧。

## 7. 时间戳和真实时间来源混淆

### 现象

早期使用：

```text
APP_LOOP_PERIOD_MS = 50
```

假设主循环每圈 50ms。实际测试发现计时不准。

### 修复

后续改用 TIM2 中断维护毫秒 tick：

```text
TIM2_IRQHandler -> App_RealMsTick++
```

### 关键理解

STM32 可以自己计时，但不能凭空知道现实世界的年月日时分秒。  
真实时间必须来自：

- PC 上位机
- 手机
- RTC + 纽扣电池
- 网络授时
- GPS
- 其他外部时间源

## 8. MPU6050 检测逻辑调参困难

### 现象

曾遇到：

- 轻微晃动不触发
- MOVE 不好触发
- 持续倾斜只触发 MOVE
- 明显冲击不触发 SHOCK
- `NONE / MOVE / NONE / MOVE` 抖动

### 根因

真实动作不是干净的。一次动作可能同时影响：

- 加速度
- 陀螺仪
- Pitch / Roll

### 修复

加入：

- 基准校准
- 阈值判断
- 连续命中确认
- 连续恢复释放
- 事件优先级
- 事件升级

### 收获

传感器项目不是阈值写死就完事。  
必须看原始数据、做边界测试、记录误报漏报。

## 9. W25Qxx 日志系统从能写到可信

### 现象

日志运行时能看到，但掉电后读取异常；连续写入多条后，第 7 条附近容易出现异常。

### 根因 1：日志恢复不稳

运行时打印 `[LOG] #n` 不代表 Flash 中的数据一定可靠。必须做写后回读校验。

### 根因 2：跨页写问题

一条日志记录约 40 字节，W25Qxx 一页为 256 字节。

第 7 条起始偏移：

```text
(7 - 1) * 40 = 240
```

写入范围：

```text
240 + 40 = 280
```

跨过 256 字节页边界。  
早期直接 `PageProgram` 整块写，导致页边界附近数据异常。

### 修复

按页分段写入：

```c
while (Length > 0)
{
    pageRemain = W25QXX_PAGE_SIZE - (Addr % W25QXX_PAGE_SIZE);
    chunk = (Length < pageRemain) ? Length : pageRemain;

    W25Qxx_PageProgram(Addr, Data, chunk);

    Addr += chunk;
    Data += chunk;
    Length -= chunk;
}
```

### 收获

Flash 不是普通数组。持久化数据必须考虑：

- 擦除
- 写使能
- 页边界
- 忙状态
- 写后校验
- 掉电恢复

## 10. 版本基线不稳

### 现象

本地代码已经修改，但对话里仍按旧文件分析，导致信息差。

### 修复

后续约定：

- 大改前先 `git status`
- 稳定后 `git add + git commit`
- 继续分析前确认当前基准文件
- 必要时重新贴当前代码
- 每次改完输出版本状态摘要

### 收获

版本管理不是最后才做的包装，而是调试过程的一部分。

## 11. 总结

这个项目最大的收获，不是单独学会某个外设，而是开始建立嵌入式系统工程思维。

核心提升包括：

- 需求边界控制
- 硬件链路分层排查
- 多文件工程组织
- 串口协议设计
- 传感器阈值和状态机调试
- Flash 日志可靠性设计
- 掉电保存和恢复
- 上位机分帧解析
- Git 版本管理
- README 和调试文档整理
