## 1. 简历精简版

基于 STM32F103C8T6 设计并实现多传感器事件黑匣子异常记录仪，完成 MPU6050 姿态采集、MOVE/TILT/SHOCK 事件检测、W25Qxx Flash 异常日志掉电保存、HC-05 蓝牙串口通信和 Python 上位机监控。项目中重点解决了串口协议升级、Python 串口接收裂包、W25Qxx 跨页写导致日志损坏、日志掉电恢复和传感器检测抖动等问题，形成了“采集 -> 检测 -> 记录 -> 掉电保存 -> 蓝牙传输 -> 上位机显示 -> 调试复盘”的完整嵌入式工程闭环。

## 2. 简历条目版

**Blackbox：基于 STM32 的多传感器事件黑匣子异常记录仪**

- 基于 STM32F103C8T6 搭建异常事件记录系统，使用 MPU6050 采集加速度与陀螺仪数据，实现 MOVE/TILT/SHOCK 事件检测。
- 设计 W25Qxx Flash 日志模块，支持异常事件写入、掉电保存、最后一条日志读取和日志清空。
- 修复 W25Qxx 页边界跨页写导致日志损坏的问题，将 PageProgram 改为按页分段写入，并增加写后回读校验。
- 基于 USART1 + HC-05 实现蓝牙串口通信，设计 PING/GET/LAST/CLR/HELP/TIME 文本协议。
- 使用 Python + Tkinter 实现 PC 上位机，支持状态显示、串口日志、自动刷新 GET、EVT 异步事件上报。
- 解决串口接收裂包、检测逻辑抖动、日志掉电恢复和多文件工程构建等问题，完成从采集到上位机显示的工程闭环。

## 3. 面试讲解版

这个项目是一个基于 STM32F103C8T6 的多传感器事件黑匣子系统，目标是在设备发生移动、倾斜、冲击等异常时，能够检测事件、记录日志，并通过蓝牙上位机查看当前状态和历史记录。

我主要完成了 MPU6050 数据采集与事件检测、W25Qxx Flash 日志持久化、USART1/HC-05 蓝牙通信协议和 Python 上位机显示。开发过程中不只是调通外设，还重点解决了几个工程问题：

第一，早期串口命令是单字节形式，后续为了支持蓝牙和上位机交互，升级为基于 `\r\n` 的文本行协议。

第二，Python 串口接收存在裂包问题，最终通过接收缓冲区和按行分帧解决。

第三，W25Qxx 日志系统中，一条记录约 40 字节，第 7 条从页偏移 240 开始写，跨过 256 字节页边界，直接 PageProgram 整块写会导致日志损坏，后续改为按页分段写入并加入回读校验。

第四，MPU6050 检测逻辑从单帧阈值判断升级为带连续确认、释放条件和事件优先级的状态机。

这个项目让我系统训练了嵌入式项目中的硬件接线、模块划分、协议设计、Flash 持久化、上位机联调、调试证据链和 GitHub 项目包装能力。

## 5. 面试高频追问准备

### Q1：你这个项目最有价值的 bug 是什么？

W25Qxx 跨页写问题。一条记录约 40 字节，第 7 条从偏移 240 写入，跨过 256 字节页边界。早期直接 PageProgram 整块写，导致记录损坏。后续我改为按页分段写入，并增加写后回读校验。

### Q2：为什么不能直接 PageProgram 一整条记录？

因为 Flash 页编程有页边界限制。即使一条记录长度小于 256 字节，如果起始地址已经靠近页尾，也可能跨页。不能只看长度，还要看起始地址。

### Q3：为什么串口要做分帧？

串口是字节流，不是消息包。一次 read 不保证拿到一整条响应，所以必须用缓冲区累计数据，再按 `\n` 提取完整行。

### Q4：为什么检测逻辑不能只用 if-else 阈值？

真实传感器数据有噪声和抖动。单帧超过阈值就切换状态，会导致误报和状态闪烁。因此加入连续命中、连续释放和事件优先级。
